1 介绍

  1. 为什么需要编码规范
    • 一个软件的生命周期，大部分时间都是在后期的维护上面。
    • 我们写的代码，不仅仅是我们自己看，尤其维护人员也要解读。
    • 一个软件，不是单独个人开发，而是一个团队开发。

    统一的编码规范有助于，程序的易读性，理解性，维护性。

  2. 相关编码规范参考
    • 《Code Convention for the Java^TM Programming Language》

2 编码规范

2.1 命名规范

2.1.1 命名原则

  1. 使用完整的英文名命名，慎用字母缩写。
  2. 避免命名过长(30个字符内比较好）
  3. 避免相似的命名
  4. 命名最好代表具体的使用意义。

2.1.2 包命名

命名原则：采用反域名命名规范，全部使用小写字母。包名的前缀多是顶级域名，通常为com,edu,gov,mil,net,org. 二级包名多为项目名 称，三级包名多为公司，组织，团队名， 四级包名多为功能模块名称。在我们这里是：

包名说明
com.idoukou.thu.activity展示页面的Activity类，并控制业务操作。
com.idoukou.thu.model实体层，接收后台服务的数据实体
com.idoukou.thu.service业务层，负责业流程上的操作，完成与服务器端的交互
com.idoukou.thu.ui自定义的组件
com.idoukou.thu.ui.adapter自定义的适配器，完成对某些组件的条目显示
com.idoukou.thu.utils工具类集合

2.1.3 类，接口命名

命名原则：名词，尽量避免缩写。首字母大写，非首字母全部小写。类名包含多个单词时，采用驼峰表示方法。 一般接口使用able，er，ible等后缀。
例如： class Raster; class ImageSprite;

类名说明
Acitivity类Actiity后缀标识。 如欢迎迎面WellcomeActivity
Adapter类Adapter后缀标石。书籍详情适配类,BookDetailApdater
Constants类Constants标识后缀。如链接地址常量,HttpUrlConstants
公共方法类Tools或Manager后缀标识。如线程池管理类,ThreadPoolManager
数据库类DBHelper后缀标识。如书籍数据库,BookDBHelper
自定义View类以功能名词后缀标识.如CustomeListView。
Service类以Service后缀标识.如UserService

2.1.4 方法的命名

命名原则：动词或名词。第一个单词首字母小写，其后单词的首字母大写。
例如： public void run(); public String getBookName();
类中常用的方法的命名：

方法名规范写法与说明
initXXX()初始化相关方法使用init前缀
isXXX(),checkXXX()，hasXxx()返回boolean类型的方法使用is,check或has前缀标识
getXXXX()返回基本类型，或自定义类型的方法，使用get前缀标识。
findXxxx()返回列表列表属性。

2.1.5 变量命名

命名原则：第一个单词说字母小写，其后单词首字母大写，多个单词采用驼峰标识方法。类中控件名称必须与xml布局id保持一直。
例如： filePath, textTitle

  1. 常变量命名，全部大写，单词之间用下划线隔开。
    例如： static final int MIN_WIDTH = 4
  2. 尽量避免单个字符的变量名，除非是一次性的临时变量。临时变量通常取名为i,j,k,m和n,他们一般用于整型；c,d,e一般用于字 符类型。

2.1.6 项目编码统一为 UTF-8

2.2 注释规范

2.2.1 Java 代码规范

Java程序有两类注释：实现注释（implementation comments) 和文档注释(document comments). 实现注释使用/*…*/和//界定的 注释。文档注释由/**….*/界定。文档注释可以通过javadoc工具转换成HTML文件。

  1. 文件注释 所有的源文件都应该在开头有一个注释，列出类名，版本信息，日期和版本申明。
    如下： 
    /*
 * 文件名
 * 包含类名列表
 * 创建日期
 * 版本声明
 */
  2. 类注释 每个类都要包含如下格式的注释，用于说明此类的功能等。 
    /**
 * 类名
 * @author 作者 <br />
 * 实现的主要功能
 * 创建日期
 * 修改者，修改日期，修改内容
 */
  1. 方法的注释 每个方法都要包含如下格式的注释，包含当前方法的用途，当前方法的参数含义，当前方法的返回值的内容和抛出的异常列表。 
    /**
 *
 * 方法的一句话概述
 * <p>方法详述（简单方法不必详述)</p>
 * @param s 说明参数含义
 * @return 说明返回值含义
 * @throws IOException 说明发生此异常的条件
 * @throws NullPointerException 说明发生此异常的条件
 */
  2. 属性注释
    1. private 及默认访问标识修饰的类属性使用单行注释
    2. public 以及 protected 访问标识修饰的类属性使用文档注释

    正确样例： 

    private int dialogState = 0; // 对方状态 (0 不显示，1 显示)
/**
 * 最大高度
 */
public static final int MAX_WIDTH = 100;
  3. 业务变更新增代码注释 当版本确定后，新增业务代码时，需要添加注释说明,样本示例 
    1:  //@begin added by Luc at 2013-12-24
2:  .......bla bla ...code ..
3:  //@end added by Luc at 2013-12-24
  4. 业务变更修改代码注释 当版本确定后，业务变更而修改代码时，需添加注释说明，样本示例 
    //@begin updated by Luc at 2013-12-24 
....bla bla ...code ....
//@end updated by Luc at 2013-12-24
  5. 业务变更删除代码注释 当版本确定后，业务变更而需删除代码时，需添加注释说明，样本示例 
    //@begin deleted by Luc at 2013-12-24
// 删除代码行
//@end deleted by Luc at 2013-12-24

2.3 编写规范

2.3.1 三原则"对齐，分块，统一"。

  1. 对齐：变量对齐，符号对齐，注释对齐. 
    private String userName;  // 姓名
private int    age;       // 年龄

public void init(){
    userName  =  "lc";
    age       =  10;
}
  2. 分块： 相同操作的放到一块， 操作紧密的放到一块, 块之间用空行分割 
    /**
 *  Http Get请求调用
 * @param requsetUrl 请求路径
 * @return 返回解析后封装的JSONObject对象
 */
 public static JSONObject getRequst(String requestUrl){
    JSONObject jsonObject = new JSONObject();
    
    try{
        URL               url  =  new URL(requestUrl);
        HttpURLConncetion conn =  (HttpURLConnection)url.openConnection();
        
        conn.connect();                              //step-1 connect
        
        int resultCode = conn.getResponseCode();
        if ( 200 == resultCode) {
            InputStream       is     =   conn.getInputStream();
            InputStreamReader isr    =   new InputStreamReader(is);
            BufferedReader    reader =   new BufferedReader(isr);
            StringBuffer      buffer =   new StringBuffer();
            String            line   =   null;
            
            line = reader.readLine();                //step-2 read
            while ( null != line) {
                buffer.append(line);
            }
            reader.close();
            
            String content    = buffer.toString();   //step-3 parser
                   jsonObject = new JSONObject(content);
            
            jsonObject.put("success", true);
            jsonObject.put("msg", "操作成功");                  
        }
    }catch(Exception e){
        jsonObject.put("success",false);
        jsonObject.put("msg","操作发生异常");
    }
    return jsonObject;
 }
  3. 统一：统一编码UTF-8， 统一命名规范，统一注释规范

2.3.2 缩进排版

4个空格常被用作缩进排版的一个单位，在实际的应用中，进行采用Tab进行排版对齐。

  1. 行长度 尽量避免一行长度超过108个字符，很多编辑器，IDE都超过单行能够一次显示的字符个数。
  2. 换行 当一个表达式无法容纳在一行内时，可依据下面规则进行断开之:
    • 在一个逗号后面断开
    • 在一个操作符前面断开
    • 选择较高级别的断开，非较低级别的断开
    • 新一行应该与上一行同一级别的表达式的开头对齐
    • 若是以上规则到时换行，以8空格缩进。

下面是断开方法调用的例子： 

//avoid
someMethod(longExpressoin1, longExpressoin2, longExpressoin3,
                 longExpressoin4, longExpression5);
//preffer
var = someMethod1(longExpressoin1, 
                         someMethod2( longExpression2,
                                             longExpressoin3));

下面是两个断开算术表达式的例子，前者更好，因为断开位于较高级别。 

//preffer
longName1 = longName2 * ( longName3 + longName4 - longName5)
                      + 4 * longName6;
//avoid
longName1 = longName2 * (longName3 + longName4
                                       - longName5) + 4 * longName6;

if 语句的换行通常使用8个空格，即一个Tab键，因为4个空格看起来比没对齐。 

//don't use this identation
if (( conditionl && condition2)
    ||(condition3 && condition4)  
    || !(condition5 && condition6)){
    doSomethingHere();   //make this line easy to miss.
}
//use this indentation instead.
if (( condition && condition2)
        || (condition3 && condition4)
        || !(condition5 && condition6)){
    dosomethingHere();
}

三元运算法表达式的处理 

alpha = (alongBooleanExpression) ? beta : gamma; //ok
alpha = (alongBooleanExpression) ? beta
                                 : gama;         //ok
alpha = (alongBooleanExpresson)
        ? beta
        : gamma;  //bad

2.3.3 语句

  1. 简单语句，每行至多包含一条语句 
    argv++;     //correct
argv++; argc--; //avoid!
  2. 所有的if,while, for 都不能省略{}符号。 
    if (isSent)
    return true;   //avoid
if (isSent)
{                  //avoid
    return true;
}

if (isSent){
    return true;   //ok
}
  3. getter, setter方法，可以简单的写到一行上 
    class Book{
    private String  name;   //书名
    private Integer id;     //书籍id

    public String  getName(){ return name; }
    public Integer getId()  { return id;   }
    public void setName(String name) { this.name = name; }
    public void setId(Integer id)    { this.id   = id;   } 
}
  4. 空格 下列情况应该使用空格：
    • 一个紧跟着括号的关键字应该被空格分开，例如： 
      while （true) {
     .....
 }

      注意：空格不应该置于方法名与其左括号之间。这件有助于区别关键字与方法调用。

    • 空格应该位于参数列表中的逗号的后面
    • 所有的二元运算符，处理"."，应该使用空格将之与操作数分开。一元操作符和操作数之间不应该加空格， 比如：负号（"-")、自增("++")和自减("–").例如： 
      a += c + d；
a = (a + b) / (c * d);
while( d++ = s++){
    n++;
}
    • for 语句中的表达式应该被空格分开，例如： for ( expr1; expr2; expr3)
    • 强制类型转换后应该更一个空格，例如： 
      myMethod((byte) aNum, (Object) x);

2.4 编写习惯与建议

2.4.1 约定俗成

  1. 变量赋值
    • 避免在一个语句中，对多个变量赋值
    • 不要将赋值运算与相等关系运算混淆
    • 不要使用内嵌的赋值运算符 
      d = (a = b + c) + r; //avoid
// preffer
a = b + c;
d = a + r;
    • 多使用圆括号划分层次 
      if ( a==b && c == d) //avoid
if ((a==b) && (c==d)) //ok

2.4.2 21中代码的坏味道

  1. Dulicated Code
    代码重复几乎是最常见的异味，他也是Refactoring的主要目标之一。
  2. Long method
    一个方法应当具有自我独立的意图，不要把几个意图放在一起。
  3. Large Class
    大类就是把太多的责任交给一个类了，应该采用one class one responsibility
  4. Divergent Change
    一个类里面的内容变化率不同。某些状态一个小时变一次，有些几年变一次；面向对象的 抽象就是讲相对不变的与相对变化的进行隔离。这使得相对不变的可以重用。
  5. Shotgun Surgery
    这正好和上面相反。对系统一个地方的改变涉及到其他许多地方的改变。这些变化率和变化内容相似的状态和行为通常应该放在 一个类中。
  6. Feature Envy
    对象的目的就是封装状态以及这些状态紧密相关的行为。如果一个类的方法频繁用get方法存取其他类的状态进行计算，那么你要 考虑把行为涉及到的状态数目最多的那个类。
  7. Data Clumps
    某些数据通常想孩子一样成群玩耍；一起出现在很多类的成员变量中，一起出现在许多方法的参数中…, 这些数据或许应该自己 独立形成类。
  8. Primitive Obsession
    面向对象的新手通常习惯使用几个原始类型的数据来表示一个概念。譬如对于范围，他们会使用两个数字。对于money，他们会使 用一个浮点数表示。因为你没有使用对象表达问题中的存在的概念，这使得代码难以理解，解决问题的难度大大增加。好的习惯 是扩充语言所能提供的原始类型，用小对象来表示范围、金额、转化率等等。
  9. Switch Statement
    基于常量的开关语句是OO的大敌，你应当把他变为子类，state或strategy。
  10. Parallel Inheritance Hierarchies
    并行的继承层次是shotgun surgery的特殊情况。因为当你改变一个层次中的某一个类时，你必须同时改变另外一个层次的并行 子类。
  11. Lazy Class
    一个干活不多的类。类的维护需要额外的开销。如果一个类的责任太少，就应当消除。
  12. Speculative Generality
    一个类实现了从未用到的功能的通用性。通常这样的类或方法唯一的用户是testcase，不要犹豫的删除它。
  13. Temporay Field
    一个对象的属性可能只在某些情况下有意义。这样的代码将难以理解。专门建立一个对象来持有这样的孤儿属性，把只和 他相关的行为移动到该类。最常见的是一个特定算法需要某些这些该算法才有用的变量。
  14. Message Chain
    消息链发生在于当一个客户向一个对象要求另一个对象，然后客户又向这另一对象要求另一个对象，…. 如此…,这时，需要隐藏分派。
  15. Middle Man
    对象的基本特性之一就是封装，而你经常会通过分派去实现分装。但是这一步不能走的太远，如果你发现 一个接口的一大半方法都在做分派，你可能需要移去这个中间人。
  16. Inappropriate Intimacy
    某些类相互之间太亲密，他们花费了太多的时间去专研对方的私有部分。对人类而言，我们也许不应该太假正经了 ，但我们应该让自己的类严格遵守禁欲主义。
  17. Alternative Classes with Different Interfaces
    做相同事情的方法不同的函数signature,一致把他们往类层次上移，直至协议一致。
  18. Incomplete Library Class
    要建立一个好的类库非常困难。我们大量的工作都是基于类库实现的，然后如此广泛而又 相异的目标对库构建者提出了苛刻的要去。库构建者也不是万能的。有时候我们会发现库 类无法实现我们的需求功能。而直接对库类的修改非常困难。这时候就需要用各种手段进行Refactoring。
  19. Data Class
    对象包括状态和行为。如果一个类只有状态没有行为，那么肯定有什么地方出错了。
  20. Refused Bequest
    超类传下来很多行为和状态，而子类只是用了其中的一小部分。这通常以为着你的类层次有问题。
  21. Comments
    经常觉得写很多注释便是你的代码难以理解，如果是这样的，表示需要refactoring.
